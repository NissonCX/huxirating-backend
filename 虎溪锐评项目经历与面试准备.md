## 虎溪锐评（后端开发）

**时间**: 2025.07 — 2025.08　　**技术栈**: Spring Boot · MyBatis-Plus · MySQL · Redis · RabbitMQ · Redisson · Sentinel · Caffeine · Lua · Docker

**项目简介**：基于 Spring Boot 的本地生活服务平台（类大众点评），支持商户浏览、探店笔记、点赞关注、优惠券秒杀等核心功能。重点攻坚高并发场景下的秒杀系统架构，通过多层分布式技术保障数据一致性与系统高可用。

**核心职责与技术成果**：

- **秒杀全链路一致性**：设计并实现"**Sentinel 限流 → Redis + Lua 原子预扣 → RabbitMQ 异步入库 → Outbox 补偿 → DLQ 回滚**"五层架构。Lua 脚本保证库存扣减与一人一单校验的原子性，DB 层乐观锁（`stock > 0`）兜底防超卖，实现零超卖、零重复领取。
- **可靠消息传递**：基于 RabbitMQ 实现 Publisher Confirms（ConfirmCallback + ReturnsCallback + mandatory）+ 手动 ACK + 重试队列（TTL 延迟）+ 死信队列的四级消息保障机制。生产端发送失败写入 Outbox 发件箱表，定时补偿任务附带 CorrelationData 投递，Broker 异步 confirm(ack) 后才标记已发送，杜绝假成功。死信队列配置 Lazy Queue + TTL 7天 + maxLength 10000 防堆积溢出。全链路零消息丢失。
- **多级缓存架构**：构建 Caffeine（L1 本地缓存）+ Redis（L2 分布式缓存）两级缓存体系。L1 微秒级响应减少网络开销，写操作采用 Cache Aside 模式（先更新 DB → 删 Redis → 删 Caffeine）保障一致性。封装通用 `CacheClient` 工具类，支持缓存穿透（空值缓存）、缓存击穿（互斥锁重建 / 逻辑过期）、缓存雪崩（随机 TTL）三种策略。
- **限流熔断防护**：集成 Sentinel 实现三维限流：整体 QPS 限流（200/s）+ 热点参数限流（单券 50/s）+ 异常比例熔断（50% 阈值），防止恶意刷单和热点券击垮系统。
- **社交 Feed 流**：基于 Redis ZSet 实现推模式 Feed 流 + 滚动分页（`ZREVRANGEBYSCORE`），解决传统分页在动态列表中数据重复/遗漏问题。利用 Redis Set 的 `SINTER` 实现共同关注查询。
- **全局唯一 ID**：基于 Redis INCR 实现分布式 ID 生成器（1 位符号 + 31 位时间戳 + 32 位序列号），按日分 Key 便于统计，避免 ID 规律泄露业务信息。
- **容器化部署**：Docker Compose 一键编排 MySQL + Redis + RabbitMQ + App 四个服务，健康检查就绪后再启动应用，开箱即用。

## 一、项目介绍类

### Q1：请介绍一下你做的这个项目

**参考回答**：

> 这是一个基于 Spring Boot 的本地生活服务平台，类似大众点评，核心包含用户登录、商户查询、优惠券秒杀、达人探店和关注推送等功能。
>
> 技术上最复杂的模块是**秒杀优惠券**。我把传统的同步下单（校验→扣减→入库）拆成了两段：**前端链路**用 Redis + Lua 脚本做毫秒级的原子校验和预扣库存，通过后立即返回订单号；**后端链路**通过 RabbitMQ 异步完成数据库的库存扣减和订单创建。为了保证全链路的数据一致性，我设计了重试队列 + 死信队列 + Outbox 补偿三层兜底机制。入口还用 Sentinel 做了 QPS 限流、热点参数限流和熔断降级。
>
> 缓存方面，我搭建了 Caffeine + Redis 的两级缓存架构，并封装了通用的缓存工具类来解决穿透、击穿、雪崩问题。社交模块用 Redis ZSet 实现了 Feed 推模式和滚动分页，用 GEO 实现了附近商户搜索，用 Bitmap 实现了用户签到统计。
>
> 整个项目通过 Docker Compose 一键部署。

---

### Q2：你在这个项目中遇到的最大难点是什么？怎么解决的？

**参考回答**：

> 最大的难点是**秒杀场景下全链路的数据一致性**。
>
> 因为我把下单拆成了 Redis 预扣和数据库实际扣减两步，中间通过 MQ 桥接，就必须处理**消息丢失、消费失败、重复消费**三大问题。
>
> 1. **消息丢失**：生产端开启 RabbitMQ 的 Publisher Confirms + Returns 机制，通过 `ConfirmCallback` 感知消息是否到达交换机，`ReturnsCallback` + `mandatory=true` 感知消息是否路由到队列。确认失败时将消息写入 Outbox 表，定时任务每 30s 扫描补偿重发，且补偿发送附带 `CorrelationData("outbox:" + id)`，只有 Broker 异步回调 `confirm(ack=true)` 后才标记 Outbox 已发送，杜绝"发了但 Broker 没收到"的假成功。
> 2. **消费失败**：消费端使用手动 ACK（`acknowledge-mode: manual`），业务成功才 `basicAck`。异常后消息进入重试队列（TTL=5s 后回到主队列），最多重试 3 次；仍然失败则进入死信队列，由 DeadLetterConsumer 先做幂等校验（orderId 查重防重复回滚），再回滚 Redis 数据（恢复库存 + 移除用户记录），最后落库取消订单。死信消费也使用手动 ACK，处理失败时 `basicNack(requeue=false)` 避免死循环，记录日志人工介入。
> 3. **重复消费**：消费前先查数据库判断订单是否已存在（幂等校验），同时加 Redisson 分布式锁按 userId 串行，双重保障。
>
> 另外一个难点是多级缓存的一致性。Caffeine 是进程内缓存，更新 DB 后需要同时删除 Redis 和 Caffeine。我采用"先更新 DB → 删 Redis → 删 Caffeine"的顺序，配合短 TTL 兜底，在单机部署场景下保证了可接受的一致性窗口。

## 二、秒杀系统深度拷打

### Q3：你的秒杀方案如何保证不超卖？

**参考回答**：

三层联防：

| 层级 | 手段 | 原理 |
|------|------|------|
| **Redis 层** | Lua 脚本原子扣减 | `GET` 库存 → `SISMEMBER` 校验资格 → `INCRBY -1` 扣减，三步在一个 Lua 脚本中原子执行，不会出现读-改-写的竞态 |
| **DB 层** | 乐观锁 `WHERE stock > 0` | 即使 Redis 扣减成功但 DB 实际库存不足，`UPDATE` 影响行数为 0，不会扣成负数 |
| **分布式锁** | Redisson 锁 `lock:order:{userId}` | 保证同一用户在并发场景下只有一个请求能进入扣减逻辑 |

**追问**：为什么不用 `WHERE stock = #{oldStock}`（CAS）做乐观锁？

> CAS 在高并发下成功率极低，100 个请求只有一个能成功（ABA 问题且无意义重试），而 `stock > 0` 只要库存还有就能成功，吞吐量远高于 CAS。

---

### Q4：Lua 脚本具体做了什么？为什么要用 Lua？

**参考回答**：

```lua
-- 1. 判断库存是否充足
if tonumber(redis.call('get', stockKey)) <= 0 then return 1 end
-- 2. 判断用户是否已下单（一人一单）
if redis.call('sismember', orderKey, userId) == 1 then return 2 end
-- 3. 扣减库存 + 记录用户
redis.call('incrby', stockKey, -1)
redis.call('sadd', orderKey, userId)
return 0
```

**为什么用 Lua**：Redis 是单线程模型，Lua 脚本在 Redis 中是原子执行的，整个脚本执行期间不会有其他命令插入。如果用多个 Redis 命令（GET → SISMEMBER → INCRBY → SADD），在并发场景下两个请求可能交叉执行，导致同时读到库存=1 然后都扣减。

**追问**：Lua 脚本的参数是怎么传的？

> 通过 `KEYS` 和 `ARGV` 传入。`KEYS[1]` 是库存 Key（`seckill:stock:{voucherId}`），`KEYS[2]` 是订单 Key（`seckill:order:{voucherId}`），`ARGV[1]` 是 `userId`。这样遵循了 Redis Cluster 的 Key 规范。

---

### Q5：如果 Redis 预扣成功了，但 MQ 发送失败了怎么办？

**参考回答**：

> 这是一个关键的边界场景。我设计了 Outbox 模式来应对：
>
> 1. MQ 发送失败时，将订单消息序列化后写入 `tb_message_outbox` 表（status=0, retry_count=0）。
> 2. 定时任务 `@Scheduled(fixedDelay = 30000)` 每 30s 扫描 status=0 且 retry_count < 5 的记录，重新发送到 MQ。
> 3. **关键点**：补偿发送时附带 `CorrelationData("outbox:" + outboxId)`，**不再发完就标记已发送**，而是由 `RabbitMQConfirmConfig` 中注册的 `ConfirmCallback` 异步回调——只有 Broker 返回 `confirm(ack=true)` 后，才将 outbox 状态更新为 status=1（已发送）。这杜绝了"调用 `convertAndSend` 没抛异常但消息实际没到 Broker"的假成功场景。
> 4. 发送异常 retry_count++；超过 5 次则 status=2（标记永久失败），需人工介入处理。
>
> 这样即使 MQ 完全不可用，消息也不会丢失，系统最终能恢复一致性。

**追问**：Outbox 表和业务操作是在同一个事务里吗？

> 不在。因为 Redis Lua 执行和发 MQ 都不是 DB 操作，写 Outbox 是在 MQ 发送失败的 catch 块里单独写入的。这里的一致性依赖于：Redis 已经预扣成功（用户看到的是成功），Outbox 补偿保证消息最终能送达 MQ 并被消费。

### Q6：死信队列里的消息怎么处理？

**参考回答**：

> DeadLetterConsumer 监听死信队列 `order.dlq`，使用**手动 ACK 模式**，执行三步操作：
>
> **1) 幂等校验** — 先用 orderId 查库，如果订单已存在则直接 `basicAck` 跳过，防止重复回滚。
>
> **2) Redis 数据回滚**：
> ```java
> // 恢复库存
> stringRedisTemplate.opsForValue().increment("seckill:stock:" + voucherId);
> // 移除用户下单记录
> stringRedisTemplate.opsForSet().remove("seckill:order:" + voucherId, userId.toString());
> ```
>
> **3) 落库 + ACK**：创建 status=4（已取消）的订单记录 → 清除 Redis PENDING 状态 → `channel.basicAck()`。
>
> **异常处理**：如果死信消费本身也失败，使用 `channel.basicNack(deliveryTag, false, false)`——**不 requeue**。因为 DLQ 是最后一道防线，如果 requeue 会导致死循环（消费→失败→回到DLQ→消费→失败…）。此时仅记录 `log.error` 告警，由人工介入处理。
>
> 本质上是"**补偿事务**"的思路——正向流程走不通，就走逆向回滚。

**追问**：回滚 Redis 之后用户会看到什么？怎么做才能让用户体验更好？

> 如果只回滚 Redis 不做其他处理，用户拿着 orderId 查询会发现"订单不存在"——体验很差。我实现了一套 **"Pending Order + 失败订单落库"** 方案来解决：

> **1. 全程可查的订单状态追踪**
>
> | 阶段 | Redis `order:status:{orderId}` | 数据库 `tb_voucher_order` | 用户查询结果 |
> |------|------|------|------|
> | Lua 预扣成功，MQ 投递后 | `PENDING`（TTL 30min） | 无记录 | "订单处理中，请稍后查询" |
> | 消费者成功写库 | 删除 key | status=1（未支付） | 正常订单 |
> | DLQ 回滚 | 删除 key | status=4（已取消） | "下单失败，库存扣减异常，请重试" |
>
> **2. 代码实现要点**
>
> ```java
> // ① seckillVoucher() — Lua 成功后写 PENDING 状态
> stringRedisTemplate.opsForValue().set(
>         ORDER_STATUS_KEY + orderId, "PENDING", 30, TimeUnit.MINUTES);
>
> // ② OrderMessageConsumer — 落库成功后清除 PENDING
> stringRedisTemplate.delete(ORDER_STATUS_KEY + orderId);
>
> // ③ DeadLetterConsumer — 回滚 Redis 后创建「已取消」订单记录
> VoucherOrder failedOrder = new VoucherOrder();
> failedOrder.setId(orderId);
> failedOrder.setStatus(4);  // 已取消
> voucherOrderService.save(failedOrder);
> stringRedisTemplate.delete(ORDER_STATUS_KEY + orderId);
>
> // ④ 新增订单查询接口 GET /voucher-order/{orderId}
> // 先查 DB → 有则返回订单状态
> // 没有 → 查 Redis PENDING → 返回"处理中"
> // 都没有 → 返回"订单不存在"
> ```
>
> **3. 为什么「失败订单也落库」？**
>
> - **用户侧**：查询永远有结果，不会出现"我明明下了单却查不到"的困惑。status=4 的含义是"已取消"，配合 message 字段告知原因。
> - **运营侧**：失败订单记录可用于统计秒杀失败率、排查异常原因，比只打日志更方便。
> - **幂等安全**：消费者中已有 `getById(orderId)` 幂等校验，失败订单落库后即使消息重复投递也不会重复处理。
>
> **4. 本质思路**
>
> 这是**"异步操作可观测性"**的最佳实践——把异步黑盒（用户不知道后台在干什么）变成白盒（每个阶段都有明确状态）。类比电商下单后看到的"待处理 → 待支付 → 已取消"全流程状态机。

### Q7：秒杀接口的限流是怎么做的？为什么要做热点参数限流？

**参考回答**：

> Sentinel 做了三层防护：
>
> 1. **整体 QPS 限流**：秒杀接口整体限制 200 QPS，超出直接返回"系统繁忙"。
> 2. **热点参数限流**：按 `voucherId` 维度限流，单个券种 50 QPS。**原因**：如果平台同时有 10 种券在秒杀，默认只配整体限流 200/s，那么一个爆款券的流量可能独占 180/s，导致其他券种几乎无法被抢到。热点参数限流保证每种券公平分配流量。还支持例外配置，比如 voucherId=1 的特殊活动券可以放宽到 100/s。
> 3. **熔断降级**：10s 内异常比例超 50% 触发熔断，持续熔断 10s，防止下游连锁故障。

**追问**：`blockHandler` 和 `fallback` 有什么区别？

> `blockHandler` 处理**限流/熔断**触发时的回调，参数列表需要额外加一个 `BlockException`；`fallback` 处理**业务逻辑本身**抛出的异常。两者可以同时存在，优先匹配 `blockHandler`。

---

### Q8：消费端的幂等性是怎么保证的？

**参考回答**：

双重保障：
1. **查询去重**：消费消息前先用 `orderId` 查数据库，如果订单已存在直接 `return`（ACK 确认）。
2. **Redisson 分布式锁**：按 `userId` 加锁（`lock:order:{userId}`），保证同一用户的多条消息只能串行消费。

**追问**：为什么不用数据库唯一索引来保证幂等？

> 也可以配合使用。`tb_voucher_order` 的主键就是全局唯一的 orderId（由 RedisIdWorker 生成），天然不会重复插入。但仅靠唯一索引会抛异常然后触发重试，不如前置查询拦截来得高效。

## 三、缓存与一致性深度拷打

### Q9：你是怎么解决缓存穿透的？

**参考回答**：

> **缓存空对象**。当数据库查不到数据时，在 Redis 中写入一个空值（`""`），并设置较短的 TTL（2 分钟）。下次查询同一个不存在的 ID 时直接命中缓存返回空，不再打到数据库。

**追问**：如果攻击者用大量不同的不存在 ID 刷怎么办？

> 空值缓存只能应对固定 ID 的穿透。对于随机 ID 攻击，可以：
> 1. **布隆过滤器**前置拦截：将所有合法 ID 加入布隆过滤器，请求先过过滤器。
> 2. **限流**：对异常流量在 Gateway 层限流。
> 3. **ID 格式校验**：先校验 ID 的合法性（如长度、格式），不合法直接拒绝。

---

### Q10：缓存击穿你提到了两种方案，在生产中选哪种？

**参考回答**：

| 维度 | 互斥锁 | 逻辑过期 |
|------|--------|---------|
| 一致性 | 强一致 | 短暂不一致（重建期间返回旧数据） |
| 性能 | 锁等待，QPS 受限 | 无锁，QPS 高 |
| 死锁风险 | 有（需设过期时间兜底） | 无 |
| 适用场景 | 数据一致性要求高（如库存） | 允许短暂不一致（如店铺详情） |

> **生产建议**：店铺信息这种允许秒级不一致的场景用逻辑过期（性能优先）；涉及资金、库存的场景用互斥锁（一致性优先）。

---

### Q11：Cache Aside 模式为什么是"先更新 DB 再删缓存"而不是"先删缓存再更新 DB"？

**参考回答**：

> **先删缓存再更新 DB** 会出问题：
> 1. 线程 A 删除缓存
> 2. 线程 B 读缓存未命中 → 查数据库（旧值）→ 写入缓存
> 3. 线程 A 更新数据库（新值）
> 4. 结果：缓存里是旧值，数据库是新值 → **永久不一致**（直到缓存过期）
>
> **先更新 DB 再删缓存** 的不一致窗口极短（仅在缓存过期且并发读写同时发生时），配合 TTL 兜底几乎不会出问题。

**追问**：删缓存失败了怎么办？

> 1. **重试**：将删除操作发到 MQ 异步重试。
> 2. **延迟双删**：更新 DB 后先删一次缓存，延迟几百毫秒再删一次。
> 3. **TTL 兜底**：所有缓存 Key 都设置过期时间，即使删除失败，过期后也会自动清除。
> 本项目用的是"立即删除 + TTL 兜底"方案。

---

### Q12：多级缓存的一致性是怎么保证的？Caffeine 是本地缓存，多实例部署怎么办？

**参考回答**：

> 当前方案是**单实例部署**场景下的设计：写操作按"更新 DB → 删 Redis → 删 Caffeine"顺序执行，配合 Caffeine 的短 TTL（5 分钟写后过期、2 分钟访问后过期）兜底。
>
> **多实例场景**下 Caffeine 无法跨进程通知。有以下方案：
> 1. **Redis Pub/Sub**：更新数据时发布消息到 Redis 频道，各实例订阅频道后清除本地缓存。
> 2. **Canal 监听 binlog**：中间件感知 DB 变更后广播通知所有实例。
> 3. **降低 Caffeine TTL**：缩短到秒级，靠自然过期保证最终一致性（简单但有延迟）。

## 四、Redis 八股深入

### Q13：Redis 为什么快？

- **纯内存操作**：数据存储在内存中，读写速度为纳秒/微秒级
- **单线程模型**：避免了上下文切换和竞态条件（注意：6.0 后 I/O 多线程，但命令执行仍然单线程）
- **IO 多路复用**：epoll 模型，单线程监听大量连接
- **高效数据结构**：SDS、跳表、压缩列表、整数集合等针对不同场景优化

---

### Q14：Redis 的持久化机制有哪些？各有什么优缺点？

| 维度 | RDB（快照） | AOF（追加日志） |
|------|------------|----------------|
| 原理 | fork 子进程，将内存数据写入二进制文件 | 将每条写命令追加到日志文件 |
| 恢复速度 | 快（直接加载二进制） | 慢（需要重放所有命令） |
| 数据安全 | 可能丢失最后一次快照后的数据 | 最多丢失 1s 数据（everysec 策略） |
| 文件大小 | 小（压缩二进制） | 大（文本命令，需要定期 rewrite） |
| CPU 开销 | fork 时有压力 | rewrite 时有压力 |

> **生产推荐**：RDB + AOF 混合使用（Redis 4.0+），AOF rewrite 时先写 RDB 内容再追加增量命令。

---

### Q15：Redis 的内存淘汰策略有哪些？你的项目用哪种？

八种策略，关键的几种：
- `noeviction`：不淘汰，内存满了直接报错
- `allkeys-lru`：从所有 Key 中淘汰最近最少使用的
- `volatile-lru`：从设置了过期时间的 Key 中淘汰 LRU
- `allkeys-lfu`：从所有 Key 中淘汰最不经常使用的（Redis 4.0+）

> 本项目适合用 `allkeys-lru`，因为缓存数据都可以从 DB 重建。

---

### Q16：Redisson 分布式锁的实现原理？和你自己写的 SimpleRedisLock 有什么区别？

| 维度 | SimpleRedisLock | Redisson |
|------|----------------|---------|
| 加锁 | `SETNX key UUID:threadId` + `EXPIRE` | Hash 结构 + Lua 脚本（可重入） |
| 解锁 | Lua 脚本（判断值 → 删除） | Lua 脚本（判断 + 计数器递减） |
| 可重入 | 不支持 | 支持（Hash field 记录 count） |
| 自动续期 | 不支持（需手动设过期时间） | 看门狗每 10s 续期（默认 30s 过期） |
| 释放安全 | UUID + ThreadId 防误删 | Hash field + Lua 原子操作 |
| 主从一致 | 可能主从切换导致锁失效 | 支持 RedLock（多主）|

**看门狗机制**：加锁时不指定 `leaseTime`，Redisson 启动一个 TimerTask，每 `lockWatchdogTimeout / 3`（默认 10s）自动延长锁过期时间到 30s，直到业务完成调用 `unlock()`。

**追问**：看门狗会不会永远续期导致死锁？

> 不会。看门狗是以 Redisson 实例为生命周期的。如果应用宕机/重启，定时任务随进程销毁，锁到期后自动释放。

---

### Q17：Redis 的 ZSet 底层数据结构是什么？

> **跳表（skiplist）+ 哈希表**。
> - 跳表支持有序遍历（`ZRANGEBYSCORE`、`ZREVRANGEBYSCORE`），时间复杂度 O(logN)
> - 哈希表支持 O(1) 的成员查找（`ZSCORE`）
> - 当元素少且小时（ziplist/listpack 阈值），用压缩列表节省内存

### Q18：你的全局 ID 生成器为什么不用 UUID 或雪花算法？

> | 方案 | 缺点 |
> |------|------|
> | UUID | 36 字符太长，无序，不适合做 MySQL 主键（B+ 树频繁分裂） |
> | Snowflake | 依赖时钟，存在时钟回拨问题；需要配置 workerId |
> | **Redis INCR** | 简单可靠，天然有序，按日分 Key 方便统计 |
>
> 本项目的 ID 结构：1 位符号位 + 31 位时间戳（基于 2025-01-01 epoch，可用约 68 年）+ 32 位序列号（单日最大 42 亿订单），对于一个中等规模应用完全够用。

## 五、RabbitMQ 八股深入

### Q19：RabbitMQ 如何保证消息不丢失？

消息从生产到消费经过三个阶段，每个阶段都可能丢失：

| 阶段 | 丢失原因 | 解决方案 | 本项目实现 |
|------|---------|---------|-----------|
| **生产端 → Broker** | 网络故障、交换机不存在、路由失败 | Publisher Confirms + Returns | `publisher-confirm-type=correlated` + `ConfirmCallback` 异步回调确认；`publisher-returns=true` + `mandatory=true` + `ReturnsCallback` 感知路由失败 |
| **Broker 存储** | 服务宕机、内存溢出 | 交换机/队列/消息持久化 + Lazy Queue | `durable=true` + `.lazy()` 模式（消息尽早落盘，减少 RAM 占用） |
| **Broker → 消费端** | 消费失败后消息丢失 | 手动 ACK + 重试 + 死信兜底 | `acknowledge-mode: manual`；主队列消费失败 → 投重试队列成功才 ACK，投递也失败则 `NACK+requeue`；DLQ 消费失败 → `NACK(requeue=false)` 防死循环 |

> **生产端额外保障**：
> - 首次发送附带 `CorrelationData("order:" + orderId)`，可在 `ConfirmCallback` 中追踪。
> - 发送异常时写入 Outbox 表，定时补偿附带 `CorrelationData("outbox:" + outboxId)`，**只有 `confirm(ack=true)` 回调后才标记 Outbox 已发送**，而非发完就标记——杜绝假成功。
>
> **DLQ 额外保障**：
> - 配置 `TTL=7天` + `maxLength=10000` + `Lazy Queue`，防止死信堆积导致磁盘溢出。
> - DLQ 消费加入幂等校验（orderId 查重），防止重复回滚。

---

### Q20：你的重试队列是怎么实现延迟重试的？

> 利用 RabbitMQ 的 **TTL + 死信路由**：
>
> 1. 消费失败后，给消息头 `x-retry-count` 加 1，发送到 `order.retry.queue`。
> 2. `order.retry.queue` 配置了 `x-message-ttl=5000`（5 秒）和 `x-dead-letter-exchange=order.exchange`。
> 3. 消息在重试队列等待 5 秒后，自动被路由回 `order.exchange` → `order.queue`，实现延迟重试。
> 4. 重试超过 3 次后，消息发送到 `order.dlx.exchange` → `order.dlq`（最终死信队列）。

**追问**：为什么不用 RabbitMQ 的延迟队列插件？

> 延迟队列插件（`rabbitmq_delayed_message_exchange`）需要额外安装，且在消息量大时性能不如 TTL + DLX 方案。TTL + DLX 是 RabbitMQ 原生支持的，更可靠。对于固定延迟时间的场景（如本项目的 5 秒重试），TTL + DLX 完全够用。

---

### Q21：RabbitMQ 的消息确认机制（ACK）有哪几种模式？

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| `none` | 消息投递后立即确认 | 日志等允许丢失的场景 |
| `auto` | Spring 默认，无异常自动 ACK，有异常自动 NACK | 简单业务 |
| `manual` | 手动调用 `channel.basicAck()` | 需要精确控制确认时机（**本项目采用**） |

> 本项目使用 `manual`，消费方法签名注入 `Channel` 参数，在消费逻辑全部成功后才手动 `channel.basicAck(deliveryTag, false)`。
>
> **失败处理的 ACK 策略特别关键**：
> - **主队列消费失败**：先尝试投递重试/死信队列，投递**成功后才 ACK 原消息**（而非 NACK+requeue），避免消息在主队列无限循环。如果投递重试队列也失败，则 `basicNack(deliveryTag, false, true)` 让消息回到主队列等待下次消费。
> - **死信队列消费失败**：`basicNack(deliveryTag, false, false)`——**不 requeue**，因为 DLQ 是最后一道防线，requeue 会死循环。
>
> **注意**：全局配置 `acknowledge-mode: manual` 后，**所有队列的消费者都必须手动 ACK**（包括 DeadLetterConsumer），否则消息永远不会被确认，导致无限重复投递。

## 六、MySQL 与数据一致性

### Q22：乐观锁和悲观锁的区别？你的项目为什么用乐观锁？

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 思想 | 假设一定会冲突，先加锁再操作 | 假设不会冲突，提交时检查 |
| 实现 | `SELECT ... FOR UPDATE` | `WHERE stock > 0`（或版本号） |
| 性能 | 低（锁等待、死锁风险） | 高（无锁，冲突时重试） |
| 适用场景 | 写多读少、冲突严重 | 读多写少、冲突较少 |

> 秒杀场景下大量请求并发扣库存。悲观锁（`FOR UPDATE`）会让所有请求排队等锁，严重拖慢响应。乐观锁 `WHERE stock > 0` 无需加锁，数据库利用行锁保证原子 UPDATE，失败的请求直接返回"库存不足"，符合业务语义。

---

### Q23：事务的 ACID 和隔离级别？

**ACID**：原子性（Atomicity）、一致性（Consistency）、隔离性（Isolation）、持久性（Durability）

**四种隔离级别**（MySQL 默认 Repeatable Read）：

| 级别 | 脏读 | 不可重复读 | 幻读 |
|------|------|-----------|------|
| Read Uncommitted | ✓ | ✓ | ✓ |
| Read Committed | × | ✓ | ✓ |
| Repeatable Read | × | × | ✓（InnoDB 通过 MVCC+间隙锁基本解决） |
| Serializable | × | × | × |

---

### Q24：MySQL 的索引结构？为什么用 B+ 树？

> InnoDB 使用 **B+ 树**作为索引结构：
> - **非叶子节点只存 Key**，不存数据，一个页（16KB）能存更多索引项 → 树高更低 → IO 次数更少
> - **叶子节点形成双向链表**，范围查询高效（`BETWEEN`、`ORDER BY`）
> - **所有数据都在叶子节点**，查询路径长度一致，性能稳定
>
> 相比 B 树：B 树所有节点都存数据，相同数据量下树更高，IO 更多。
> 相比 Hash：Hash 等值查询 O(1)，但不支持范围查询和排序。
> 相比跳表：B+ 树磁盘友好（一个节点 = 一个磁盘页），跳表更适合内存场景（如 Redis ZSet）。

## 七、Spring 与框架八股

### Q25：Spring Boot 自动配置的原理？

> 1. `@SpringBootApplication` 包含 `@EnableAutoConfiguration`
> 2. `@EnableAutoConfiguration` 通过 `@Import(AutoConfigurationImportSelector.class)` 加载
> 3. `AutoConfigurationImportSelector` 读取 `META-INF/spring.factories`（或 Spring Boot 3.x 的 `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports`）中的全类名列表
> 4. 通过 `@ConditionalOnClass`、`@ConditionalOnBean` 等条件注解，按需加载配置类

---

### Q26：AOP 的实现原理？你的项目哪里用到了 AOP？

> Spring AOP 基于**动态代理**：
> - 目标类实现了接口 → JDK 动态代理（基于反射）
> - 目标类没有接口 → CGLIB（基于字节码继承）
>
> 本项目中 AOP 的应用：
> 1. **`@Transactional`**：Spring 通过 AOP 拦截被注解标记的方法，在方法执行前开启事务，成功提交，异常回滚。
> 2. **`@SentinelResource`**：Sentinel 通过 AOP 拦截方法调用，执行限流/熔断检查。
> 3. **拦截器链**：虽然不是 AOP，但思想类似——通过 `HandlerInterceptor` 在请求前后做登录校验和 Token 刷新。

---

### Q27：`@Transactional` 失效的场景有哪些？

1. **方法不是 public**：Spring AOP 基于代理，非 public 方法无法被代理
2. **自调用**：同一个类中方法 A 调用方法 B（B 上有 `@Transactional`），B 的事务不会生效，因为是 `this.B()` 而不是通过代理调用
3. **异常被 catch 吞掉**：事务拦截器检测不到异常，不会回滚
4. **抛出非 RuntimeException**：默认只对 `RuntimeException` 和 `Error` 回滚，受检异常需要 `rollbackFor = Exception.class`
5. **类没被 Spring 管理**：没有 `@Service`、`@Component` 等注解

---

### Q28：ThreadLocal 的原理？你的项目怎么用的？有什么问题要注意？

> **原理**：每个 Thread 内部有一个 `ThreadLocalMap`（key 是 ThreadLocal 弱引用，value 是实际存储值）。调用 `threadLocal.set(value)` 时，存到当前线程的 map 中。
>
> **本项目用法**：`UserHolder` 工具类使用 `ThreadLocal<UserDTO>` 存储当前登录用户信息。`RefreshTokenInterceptor` 在 `preHandle` 中从 Redis 读 Token → 解析用户 → `UserHolder.saveUser(userDTO)`，后续整个请求链路通过 `UserHolder.getUser()` 获取当前用户。
>
> **注意事项**：
> 1. **内存泄漏**：必须在请求结束时（`afterCompletion`）调用 `UserHolder.removeUser()` 清除，否则线程池复用线程时会读到上一个请求的用户信息。
> 2. **线程池传递**：异步任务（如 `@Async`、线程池）无法继承父线程的 ThreadLocal，需要用 `InheritableThreadLocal` 或手动传递。

### Q30：如果 Redis 宕机了，你的秒杀系统会怎样？怎么降级？

## 九、项目细节追问

### Q32：你的登录方案是怎么实现的？为什么不用 JWT？

> **当前方案**：手机号 + 短信验证码登录。验证码存 Redis（`login:code:{phone}`，TTL 2 分钟）。登录成功后生成 UUID Token，用户信息以 Hash 形式存 Redis（`login:token:{token}`，TTL 30 分钟，每次请求刷新）。Token 通过 Header `authorization` 传递。
>
> **为什么不用 JWT**：
> 1. JWT 无法主动失效（除非维护黑名单，本质又引入了状态）。
> 2. 本项目已经大量使用 Redis，Token 存 Redis 方案简单且灵活。
> 3. Redis Token 方案天然支持"单设备登录"、"主动踢人"等需求。
>
> **JWT 的优势**：无状态、分布式友好（无需共享 Session 存储）。如果要降低 Redis 压力、服务之间免 Token 校验，可以改用 JWT + Redis 黑名单方案。

---

### Q33：Redis GEO 底层是什么数据结构？

> GEO 底层是 **ZSet**。Redis 使用 **GeoHash** 算法将经纬度编码为一个 52 位整数作为 ZSet 的 score。GeoHash 将二维坐标映射到一维，相近的坐标 GeoHash 值也相近，因此可以利用 ZSet 的有序性进行范围查询。
>
> `GEOSEARCH key BYLONLAT lng lat BYRADIUS 5 km` 实际上是 ZSet 的范围查询 + 距离过滤。

---

### Q34：你用 Bitmap 做签到，怎么统计连续签到天数的？

> 1. `BITFIELD sign:{userId}:{yyyyMM} GET u{dayOfMonth} 0` 获取从第 1 天到今天的位数据（十进制整数）
> 2. 循环：`num & 1` 判断最低位（今天），如果是 1 则计数器 +1 → `num >>>= 1` 右移
> 3. 遇到 0 就停止（连续中断）
>
> 时间复杂度 O(天数)，最多 31 次循环，空间仅 4 字节/月/用户。

---

### Q35：你的滚动分页是怎么实现的？为什么不用传统分页？

> **传统分页的问题**：Feed 流是动态列表，新数据不断插入。用 `LIMIT offset, count` 分页，前一页的 offset 会因新数据插入而偏移，导致数据重复或遗漏。
>
> **滚动分页方案**：
> - 使用 `ZREVRANGEBYSCORE key maxScore minScore LIMIT offset count`
> - `maxScore`：第一次传当前时间戳，后续传上一页最后一条的 score
> - `offset`：上一页中与最后一条 score 相同的记录数（处理 score 重复的情况）
> - 每次查询相当于"给我 score 小于 X 的前 N 条"，不受新数据插入影响

### 2.2 关键技术点解析

#### 2.2.1 为什么拆成两段式（预扣 + 异步入库）？

传统同步方案：`校验库存 → 扣减库存 → 创建订单` 全在一个 DB 事务中完成。问题：
- MySQL 行锁导致请求串行化，QPS 极低
- 库存行成为热点行，锁竞争严重
- 响应时间 = DB 操作耗时，用户等待久

两段式方案：
- **Redis 预扣**：内存操作 + Lua 原子脚本，微秒级完成，承载高并发
- **MQ 异步入库**：消费端顺序写 DB，不受并发影响
- **用户感知**：拿到订单号 ≈ 成功，体验好（实际入库稍后完成）

#### 2.2.2 全链路数据一致性保障

这是本系统最核心的设计问题。Redis 预扣和 DB 入库本质是**跨资源的分布式事务**，不满足传统 ACID，采用**最终一致性**方案：

**一致性保障矩阵**：

| 异常场景 | 导致的不一致 | 恢复机制 | 最终结果 |
|---------|-------------|---------|---------|
| MQ 发送成功，消费成功 | 无 | — | 一致 ✓ |
| MQ 发送失败（catch 异常） | Redis 已扣，DB 未扣 | 写入 Outbox → 定时补偿重发（ConfirmCallback 确认到达后才标记已发送） | 最终一致 ✓ |
| MQ 发送成功但 confirm(nack) | Redis 已扣，DB 未扣 | ConfirmCallback 日志告警 → Outbox 保持 status=0 → 补偿重发 | 最终一致 ✓ |
| MQ 发送成功，消费失败（可恢复） | Redis 已扣，DB 未扣 | 重试队列延迟重试（最多3次），投递成功才 ACK 原消息 | 最终一致 ✓ |
| 消费失败，投递重试队列也失败 | Redis 已扣，DB 未扣 | NACK+requeue → 消息回到主队列 → 下次消费再尝试 | 最终一致 ✓ |
| MQ 发送成功，消费失败（不可恢复） | Redis 已扣，DB 未扣 | 死信队列 → 幂等校验 → Redis 回滚 + 取消订单落库 | 最终一致 ✓ |
| 死信消费也失败 | Redis 已扣，DB 未扣 | NACK(requeue=false) + 日志告警 → 人工介入 | 需人工 ⚠ |
| Outbox 补偿也失败（超5次） | Redis 已扣，DB 未扣 | 标记 status=2，人工介入 | 需人工 ⚠ |
| 应用宕机（发 MQ 前） | Redis 已扣，MQ 未发 | 重启后 Outbox 补偿扫描 | 最终一致 ✓ |

**为什么不用分布式事务（2PC / TCC）？**
- 2PC：需要所有参与者都支持 XA 协议，Redis 不支持
- TCC：实现复杂，需要为每个操作编写 Try/Confirm/Cancel 三个接口
- **本方案的最终一致性**对秒杀场景足够：用户拿到订单号后，99.9% 的情况下异步入库在毫秒内完成

#### 2.2.3 不超卖的三层防线

```
       ┌───────────────────────────────────────┐
       │         第一层：Redis Lua 脚本          │
       │  原子操作：读库存 → 判断 → 扣减         │
       │  挡住 99.9% 的超卖请求                  │
       ├───────────────────────────────────────┤
       │         第二层：DB 乐观锁               │
       │  WHERE stock > 0                       │
       │  兜底防止 Redis 与 DB 不一致导致的超卖   │
       ├───────────────────────────────────────┤
       │         第三层：Redisson 分布式锁        │
       │  lock:order:{userId}                   │
       │  防止同一用户并发消费导致的重复下单       │
       └───────────────────────────────────────┘
```

#### 2.2.4 一人一单的三层防线

| 层级 | 手段 | 说明 |
|------|------|------|
| Redis 层 | `SISMEMBER seckill:order:{id} userId` | Lua 脚本内原子判断 |
| 分布式锁 | Redisson `lock:order:{userId}` | 防并发消费 |
| DB 层 | `SELECT COUNT(*) WHERE user_id=? AND voucher_id=?` | 最终校验 |

## 三、多级缓存架构详解

### 3.1 整体架构

```
         请求到达
            │
    ┌───────▼───────┐
    │  Caffeine L1  │  ← 进程内缓存，微秒级
    │  max=1000     │
    │  writeExpire  │
    │  =5min        │
    │  accessExpire │
    │  =2min        │
    └───┬───────────┘
        │ miss
    ┌───▼───────────┐
    │   Redis L2    │  ← 分布式缓存，毫秒级
    │  TTL=30min    │
    │  + random(5)  │  ← 随机值防雪崩
    └───┬───────────┘
        │ miss
    ┌───▼───────────┐
    │    MySQL      │  ← 持久化存储
    └───┬───────────┘
        │ 查到数据
        │
    回填 Redis L2
    回填 Caffeine L1
    返回数据
```

### 3.2 读流程

```java
// 1. 查 Caffeine
Shop shop = shopCache.getIfPresent(id);
if (shop != null) return shop;

// 2. 查 Redis
String json = stringRedisTemplate.opsForValue().get("cache:shop:" + id);
if (json != null && !json.isEmpty()) {
    shop = JSON.parseObject(json, Shop.class);
    shopCache.put(id, shop);  // 回填 L1
    return shop;
}

// 3. 空值判断（缓存穿透防护）
if ("".equals(json)) return null;

// 4. 查 DB
shop = getById(id);
if (shop == null) {
    // 缓存空值，防穿透
    stringRedisTemplate.opsForValue().set(key, "", 2, TimeUnit.MINUTES);
    return null;
}

// 5. 回填缓存
stringRedisTemplate.opsForValue().set(key, JSON.toJSONString(shop),
    30 + RandomUtil.randomInt(5), TimeUnit.MINUTES);  // 随机 TTL 防雪崩
shopCache.put(id, shop);  // 回填 L1
return shop;
```

### 3.3 写流程（Cache Aside Pattern）

```java
// 1. 先更新数据库
updateById(shop);
// 2. 删除 Redis 缓存
stringRedisTemplate.delete("cache:shop:" + shop.getId());
// 3. 删除 Caffeine 本地缓存
shopCache.invalidate(shop.getId());
```

**为什么是"先更新 DB 再删缓存"而不是"更新缓存"？**
- 更新缓存在并发写时可能写入旧值（A 更新 DB → B 更新 DB → B 更新缓存 → A 更新缓存 → 缓存是 A 的旧值）
- 删除缓存更安全，下次读取时自动从 DB 加载最新值（lazy load）

### 3.4 三大缓存问题解决方案

#### 缓存穿透（查不存在的数据）

```
攻击者请求 id=-1
    │
    ▼
Redis MISS → DB MISS → 缓存空值 "" (TTL=2min)
    │
    ▼
下次请求 id=-1
Redis HIT (空值) → 直接返回 null（不打 DB）
```

#### 缓存击穿（热点 Key 过期 + 大并发）

**方案一：互斥锁重建**
```
线程A: Redis MISS → SETNX lock:shop:1 成功 → 查 DB → 写缓存 → 释放锁
线程B: Redis MISS → SETNX lock:shop:1 失败 → sleep(50ms) → 重试读缓存
线程C: Redis MISS → SETNX lock:shop:1 失败 → sleep(50ms) → 命中缓存
```

**方案二：逻辑过期**
```
线程A: 读缓存 → 发现 expireTime < now → SETNX 抢锁 → 提交异步线程重建缓存 → 返回旧数据
线程B: 读缓存 → 发现 expireTime < now → SETNX 抢锁失败 → 返回旧数据
异步线程: 查 DB → 更新缓存（新 expireTime）→ 释放锁
```

#### 缓存雪崩（大量 Key 同时过期）

- **TTL 添加随机值**：`30min + random(0~5)min`，打散过期时间
- **Redis 集群**：防止单点故障导致整体不可用
- **Caffeine 兜底**：Redis 不可用时，L1 缓存还能承担部分流量

### 4.2 Publisher Confirms + Returns 完整实现

生产端可靠性由 `RabbitMQConfirmConfig` 统一配置：

```yaml
# application.yaml
spring.rabbitmq:
  publisher-confirm-type: correlated   # 异步 confirm 回调
  publisher-returns: true              # 开启 return 回调
```

```java
@Bean
public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
    RabbitTemplate rabbitTemplate = new RabbitTemplate(connectionFactory);
    rabbitTemplate.setMandatory(true);  // 消息无法路由时触发 Return（而非静默丢弃）

    // ConfirmCallback：消息是否到达交换机
    rabbitTemplate.setConfirmCallback((correlationData, ack, cause) -> {
        String id = correlationData != null ? correlationData.getId() : "unknown";
        if (ack) {
            // Broker 已收到 → 如果是 outbox 补偿消息，标记已发送
            markOutboxSentIfNeeded(id);  // "outbox:123" → outbox.status = 1
        } else {
            log.error("消息未到达交换机: id={}, cause={}", id, cause);
            // outbox 保持 status=0，定时任务会再次补偿
        }
    });

    // ReturnsCallback：消息到达交换机但无法路由到队列
    rabbitTemplate.setReturnsCallback(returned -> {
        log.error("消息无法路由: exchange={}, routingKey={}, replyCode={}",
                returned.getExchange(), returned.getRoutingKey(), returned.getReplyCode());
        // 通常是队列配置错误，需要告警排查
    });

    return rabbitTemplate;
}
```

**发送端携带 CorrelationData**（用于 confirm 追踪）：
```java
// 首次发送（VoucherOrderServiceImpl）
CorrelationData correlationData = new CorrelationData("order:" + orderId);
rabbitTemplate.convertAndSend(ORDER_EXCHANGE, ORDER_ROUTING_KEY, messageBody, correlationData);

// 补偿发送（OrderCompensationTask）— 用 "outbox:" 前缀标识

// 注意：不再发完就标记 status=1，由 ConfirmCallback 异步标记
```

### 4.3 手动 ACK 消费（主队列 + 死信队列）

**主队列消费者**（OrderMessageConsumer）：

```java
@RabbitListener(queues = RabbitMQConfig.ORDER_QUEUE)
public void onMessage(Message message, Channel channel) throws IOException {
    long deliveryTag = message.getMessageProperties().getDeliveryTag();
    OrderMessage orderMsg = JSONUtil.toBean(new String(message.getBody()), OrderMessage.class);
    int retryCount = (int) message.getMessageProperties().getHeaders()
            .getOrDefault("x-retry-count", 0);

    try {
        // 幂等校验 + 业务逻辑（分布式锁 + DB操作）
        if (voucherOrderService.getById(orderMsg.getOrderId()) != null) {
            channel.basicAck(deliveryTag, false);  // 已存在，跳过
            return;
        }
        createVoucherOrder(orderMsg);
        channel.basicAck(deliveryTag, false);  // 业务成功，确认
    } catch (Exception e) {
        handleFailure(channel, deliveryTag, orderMsg, retryCount);
    }
}

// 失败处理：投递重试/死信成功才 ACK，否则 NACK+requeue
private void handleFailure(Channel channel, long deliveryTag,
                           OrderMessage orderMsg, int retryCount) throws IOException {
    try {
        if (retryCount < MAX_RETRY_COUNT) {
            sendToRetryQueue(orderMsg, retryCount + 1);  // 不 catch 异常
        } else {
            sendToDeadLetterQueue(orderMsg);              // 不 catch 异常
        }
        channel.basicAck(deliveryTag, false);  // 投递成功 → ACK 原消息
    } catch (Exception ex) {
        // 投递也失败 → NACK+requeue 让消息回到主队列
        channel.basicNack(deliveryTag, false, true);
    }
}
```

**死信队列消费者**（DeadLetterConsumer）：

```java
@RabbitListener(queues = RabbitMQConfig.DLQ_QUEUE)
public void onMessage(Message message, Channel channel) throws IOException {
    long deliveryTag = message.getMessageProperties().getDeliveryTag();
    OrderMessage orderMsg = JSONUtil.toBean(new String(message.getBody()), OrderMessage.class);

    try {
        // 幂等校验：订单已存在则跳过
        if (voucherOrderService.getById(orderMsg.getOrderId()) != null) {
            channel.basicAck(deliveryTag, false);
            return;
        }
        // 回滚 Redis + 落库取消订单 + 清除 PENDING
        rollbackRedisAndSaveCancelledOrder(orderMsg);
        channel.basicAck(deliveryTag, false);
    } catch (Exception e) {
        // DLQ 是最后防线，不 requeue（否则死循环），人工介入
        channel.basicNack(deliveryTag, false, false);
    }
}
```

**核心设计原则**：
- 主队列：`NACK(requeue=true)` 是兜底，正常走重试队列
- 死信队列：`NACK(requeue=false)` 是终点，不能再循环

## 五、分布式锁实现演进

### 5.1 演进路线

```
synchronized（单JVM）
    ↓ 集群部署失效
SETNX + EXPIRE（自研 SimpleRedisLock）
    ↓ 不支持可重入、无自动续期
Redisson（生产级方案）
```

### 5.2 SimpleRedisLock 实现细节

```java
// 加锁：SETNX + 过期时间（原子操作）
Boolean success = stringRedisTemplate.opsForValue()
    .setIfAbsent(key, ID_PREFIX + threadId, timeoutSec, TimeUnit.SECONDS);

// 解锁：Lua 脚本保证原子性（防止误删其他线程的锁）
// unlock.lua:
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
end
return 0
```

**为什么解锁要用 Lua？**
如果分两步（先 GET 对比，再 DELETE），在 GET 和 DELETE 之间锁可能已过期，被其他线程获取。此时 DELETE 会误删别人的锁。Lua 脚本保证判断和删除是原子的。

### 5.3 Redisson 的优势

| 特性 | 说明 |
|------|------|
| 可重入锁 | Hash 结构，field=UUID:threadId，value=重入次数 |
| 看门狗续期 | 每 10s（leaseTime/3）自动续期到 30s |
| 锁释放通知 | 等待锁的线程订阅 Redis channel，锁释放时通知唤醒 |
| 公平锁 | 支持 `getFairLock()` |
| 联锁/红锁 | 多 Redis 实例的分布式锁（RedLock 算法） |

## 六、全局唯一 ID 设计

### 6.1 ID 结构

```
┌──────┬────────────────────────────────┬──────────────────────────────────────┐
│ 0    │ 0000000 00000000 00000000 0000 │ 00000000 00000000 00000000 00000000  │
│符号位│         31位时间戳              │           32位序列号                 │
│ 1bit │    (秒级，基于2025-01-01)       │     (Redis INCR 自增)               │
└──────┴────────────────────────────────┴──────────────────────────────────────┘
```

### 6.2 实现代码

```java
public long nextId(String keyPrefix) {
    // 1. 生成时间戳
    LocalDateTime now = LocalDateTime.now();
    long timestamp = now.toEpochSecond(ZoneOffset.UTC) - BEGIN_TIMESTAMP;
    
    // 2. 生成序列号（按天分 key，防止单 key 超过 2^32）
    String date = now.format(DateTimeFormatter.ofPattern("yyyyMMdd"));
    long count = stringRedisTemplate.opsForValue()
        .increment("icr:" + keyPrefix + ":" + date);
    
    // 3. 拼接：时间戳左移32位 + 序列号
    return timestamp << 32 | count;
}
```

**设计考量**：
- **按日分 Key**：避免单 Key 的 INCR 值过大；便于按天统计订单量
- **不用自增主键**：防止 ID 规律暴露业务量；分库分表时可能冲突
- **时间戳在高位**：ID 整体趋势递增，对 B+ 树插入友好（追加写）

## 七、社交功能核心实现

### 7.1 Feed 流推模式

```
博主发布笔记 blog
       │
       ▼
查询博主所有粉丝 (tb_follow)
       │
       ▼
遍历粉丝列表:
  ZADD feed:{fansId} {timestamp} {blogId}
       │
       ▼
粉丝打开 "关注" 页面
  ZREVRANGEBYSCORE feed:{userId} {maxScore} 0 LIMIT {offset} {count}
```

**推模式 vs 拉模式**：
- **推模式（本项目）**：写扩散，发布时推给所有粉丝。读快（直接读自己收件箱），写慢（大 V 粉丝多时扇出大）
- **拉模式**：读扩散，读时拉取所有关注人的笔记。写快，读慢（需要合并多个时间线）
- **推拉结合**：大 V 用拉，普通用户用推。本项目为简化采用纯推模式

## 九、数据库表设计核心要点

### 9.1 表关系图

```
tb_user ──1:1── tb_user_info
   │
   │ 1:N
   ▼
tb_follow (user_id, follow_user_id)
   │
tb_blog ──1:N── tb_blog_comments
   │
tb_voucher ──1:1── tb_seckill_voucher
   │
   │ 1:N
   ▼
tb_voucher_order (全局唯一ID, user_id, voucher_id)
   │
tb_message_outbox (消息补偿发件箱)
   │
tb_shop ──N:1── tb_shop_type
```

### 9.2 关键设计决策

| 设计 | 原因 |
|------|------|
| `tb_voucher_order` 使用全局唯一 ID 而非自增 | 防止暴露业务量，支持分库分表 |
| `tb_voucher` 和 `tb_seckill_voucher` 1:1 拆分 | 不是所有优惠券都参与秒杀，拆分减少字段冗余 |
| `tb_user` 和 `tb_user_info` 1:1 拆分 | 高频字段（昵称/头像）与低频字段（生日/粉丝数）分离，减少行宽 |
| `tb_blog.liked` 冗余计数字段 | 避免每次查询都 `COUNT(*)`，提升查询性能 |
| `tb_message_outbox` 的联合索引 `(status, retry_count)` | 定时任务扫描条件完全匹配索引 |

## 十、Docker 部署架构

### 10.1 服务编排

```yaml
services:
  mysql:
    image: mysql:8
    ports: ["3306:3306"]
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
    volumes:
      - mysql_data:/var/lib/mysql
      - ./src/main/resources/db/hmdp.sql:/docker-entrypoint-initdb.d/init.sql

  redis:
    image: redis:7
    ports: ["6379:6379"]
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
    volumes:
      - redis_data:/data

  rabbitmq:
    image: rabbitmq:3-management
    ports: ["5672:5672", "15672:15672"]
    healthcheck:
      test: ["CMD", "rabbitmq-diagnostics", "-q", "ping"]

  app:
    build: .
    ports: ["8081:8081"]
    depends_on:
      mysql: { condition: service_healthy }
      redis: { condition: service_healthy }
      rabbitmq: { condition: service_healthy }
```

### 10.2 启动顺序保障

`depends_on` + `condition: service_healthy` 确保中间件完全就绪后再启动应用。每个服务都配置了健康检查命令，避免应用启动时连接不上中间件导致失败。
