## 1. 项目介绍模板

**面试官：请介绍一下你的这个项目。**

**参考回答：**
> 这是一个基于 Spring Boot 的虎溪锐评平台，类似大众点评。我主要负责后端核心业务的开发。
>
> 项目涵盖了**用户登录、商户查询、优惠券秒杀、达人探店、关注推送**等功能。
>
> **技术亮点方面**：
> 1. **秒杀抢购**：利用 **Redis + Lua 脚本**原子性校验，通过 **RabbitMQ** 异步下单实现最终一致性。设计了**重试队列 + 死信队列**处理消费失败，**Outbox + 定时补偿**保证消息可靠投递，**幂等设计**防止重复消费。入口使用 **Sentinel** 做 QPS 限流 + 热点参数限流 + 熔断降级。
> 2. **多级缓存**：**Caffeine(L1) + Redis(L2)** 两级缓存架构，L1 微秒级响应减少网络开销。针对穿透、击穿、雪崩设计了完整解决方案。
> 3. **社交功能**：利用 Redis **ZSet** 实现 Feed 流推模式和滚动分页。
> 4. **工程化**：**Docker Compose** 一键部署全部中间件（MySQL + Redis + RabbitMQ + App）。

## 2. 核心业务详解

### 2.1 秒杀优惠券（异步下单）

**核心逻辑**：将同步的"校验+扣减+下单"拆分为"**Redis 快速校验**"和"**MQ 异步数据库下单**"。

#### 流程步骤
1. **用户请求**：用户发起秒杀请求。
2. **Redis 校验 (Lua脚本)**：
    - 判断库存是否充足 (`get seckill:stock:{id}`)。
    - 判断是否重复下单 (`sismember seckill:order:{id} {userId}`)。
    - **校验通过**：扣减 Redis 库存 + 记录用户已下单。
    - **校验失败**：直接返回错误信息。
3. **发送 MQ 消息**：将订单信息（orderId、userId、voucherId）发送到 RabbitMQ。若 MQ 发送失败，写入 Outbox 表由定时任务补偿。
4. **返回结果**：Lua 执行成功，立即返回订单 ID（此时数据库还没写，但用户认为已成功）。
5. **异步消费**（OrderMessageConsumer）：
    - 从 RabbitMQ 消费消息。
    - **幂等校验**：查询数据库该订单是否已存在。
    - 获取 **Redisson 分布式锁**（兜底）。
    - 扣减数据库库存（乐观锁 `stock > 0`）+ 创建订单。
6. **失败处理**：
    - 消费失败进入**重试队列**（TTL 5s 后重回主队列），最多重试 3 次。
    - 超过重试次数进入**死信队列（DLQ）**，由 DeadLetterConsumer 回滚 Redis 数据。
7. **补偿机制**：定时任务每 30s 扫描 Outbox 表，重发未成功的消息。

#### 关键代码
- **Lua 脚本**（`seckill.lua`）：保证 Redis 中判断库存、校验资格、扣减库存的**原子性**。
- **RabbitMQ 拓扑**：
  - `order.exchange` → `order.queue`（主队列）
  - `order.retry.queue`（TTL=5s，死信路由回主交换机）
  - `order.dlx.exchange` → `order.dlq`（死信队列）
- **Outbox 模式**：MQ 发送失败时写入 `tb_message_outbox` 表，定时任务扫描重发。
- **Redisson 锁**：`redissonClient.getLock("lock:order:" + userId)`，在异步消费时兜底防并发。

### 2.2 商户查询缓存优化

**场景**：查询商户详情 `queryById`。

#### 缓存穿透
- **问题**：查询不存在的 ID，每次都打到数据库。
- **解决**：**缓存空对象**，设置较短 TTL（2 分钟）。

#### 缓存击穿
- **问题**：热点 Key 过期，大量并发请求瞬间打到数据库。
- **方案1：互斥锁**：`setnx` 抢锁，抢到的查库写缓存，其他休眠重试。一致性强，性能略差。
- **方案2：逻辑过期**：Key 永不过期，Value 中存 `expireTime`。发现过期后异步重建缓存，旧线程返回旧数据。性能极高，有短暂不一致。

#### 缓存雪崩
- **问题**：大量 Key 同时过期，或 Redis 宕机。
- **解决**：TTL 添加随机值；搭建 Redis 集群。

### 2.3 达人探店（Feed流）

**场景**：关注的人发了笔记，在"关注"列表看到。

#### 推模式 (Push)
- 博主发笔记时，将笔记 ID 推送到所有粉丝的收件箱（Redis **ZSet**）。
- Key: `feed:{userId}`，Value: `blogId`，Score: `timestamp`。

#### 滚动分页
- 使用 `ZREVRANGEBYSCORE` 命令，参数：`max`（上次最小时间戳）、`min`（0）、`offset`、`count`。
- 避免传统分页在动态列表中数据重复/遗漏。

---

### 2.4 附近商户 (GEO)

- 使用 Redis **GEO** 数据结构（底层 ZSet）。
- `GEOADD` 添加商户经纬度，`GEOSEARCH` 搜索半径内并按距离排序。

---

### 2.5 用户签到 (Bitmap)

- 使用 Redis **Bitmap**，Key: `sign:{userId}:{yyyyMM}`。
- `SETBIT` 签到，`BITFIELD` 获取二进制数据统计连续签到天数。
- 一个月数据仅需 4 字节。

## 3. 高频面试题 Q&A

**Q1: 你的秒杀方案如何保证不超卖？**
> 三层防护：
> 1. **Redis 层**：Lua 脚本原子性判断库存和扣减，挡住绝大部分流量。
> 2. **数据库层**：乐观锁 `where stock > 0`，利用数据库行锁保证不扣成负数。
> 3. **分布式锁**：Redisson 锁保证同一用户在同一时间只能发起一个请求。

**Q2: 为什么使用 RabbitMQ 做异步下单？**
> - 支持消息持久化、消费确认（ACK）、死信队列。
> - 通过 TTL + DLX 实现延迟重试，消费失败的消息不会丢失。
> - Publisher Confirm 机制保证消息成功投递到交换机。
> - 配合 Outbox 模式，即使 MQ 不可用也能通过定时补偿保证最终一致性。
> - 相比 Redis Stream，RabbitMQ 更成熟、生态更完善，适合生产级可靠消息传递。

**Q3: 如何保证消费的幂等性？**
> 消费前先查询数据库是否已存在该订单。如果订单已存在，直接返回成功（跳过处理），避免重复创建。同时 Redisson 分布式锁按 userId 加锁，防止并发消费。

**Q4: 消息发送失败怎么办？**
> 采用 Outbox 模式：MQ 发送失败时，将消息写入 `tb_message_outbox` 表（status=0）。定时任务每 30s 扫描未发送的消息进行重发，成功后更新状态。超过重试次数标记为失败，人工介入。

**Q5: 死信队列里的消息怎么处理？**
> DeadLetterConsumer 监听 DLQ，执行 Redis 数据回滚：恢复库存（INCR）并移除用户下单记录（SREM），保证 Redis 与数据库最终一致。同时记录日志，便于人工排查根因。

**Q6: 数据库和缓存的一致性怎么保证？**
> 采用 **Cache Aside Pattern**。读：先读缓存→未命中查库写缓存。写：**先更新数据库，再删除缓存**（Redis + Caffeine 双删）。配合缓存过期时间兜底。

**Q7: 多级缓存是怎么设计的？**
> Caffeine 作为 L1 本地缓存（微秒级），Redis 作为 L2 分布式缓存（毫秒级），DB 作为最终数据源。
> 读：L1 → L2 → DB，命中即返回并回填上层。写：先更新 DB，再删 L2（Redis），最后删 L1（Caffeine）。
> Caffeine 设置较短 TTL（5min）+ 最大容量（1000），防止本地缓存膨胀和数据不一致窗口过大。

**Q8: Sentinel 限流怎么做的？有哪些规则？**
> 三种规则：
> 1. **QPS 限流**：秒杀接口整体 QPS 上限 200，超出直接拒绝返回友好提示。
> 2. **热点参数限流**：按 voucherId 维度限流（单券 QPS 50），防止单个热门券独占所有流量。支持特殊参数值例外配置。
> 3. **熔断降级**：10s 内异常比例超 50% 触发熔断，持续 10s，期间所有请求走降级逻辑返回"系统繁忙"。
> 实现上使用 `@SentinelResource` 注解，配合 `blockHandler`（限流回调）和 `fallback`（异常降级回调）。

**Q9: Redisson 的看门狗机制是什么？**
> 业务执行时间超过锁过期时间时，看门狗每隔 10s 自动延长锁有效期，直到业务完成释放锁，防止锁提前释放导致并发问题。

**Q10: ID 生成器怎么实现的？**
> 基于 Redis INCR 的全局唯一 ID：**1位符号位 + 31位时间戳 + 32位序列号**。Key 包含日期（如 `icr:order:20231219`），方便按天统计，避免单 Key 数值过大。

## 4. 项目亮点与难点总结

**亮点：**
1. **高性能秒杀**：Sentinel 限流 → Redis + Lua 预扣 → RabbitMQ 异步入库 → Outbox 补偿 → DLQ 兜底回滚，完整的最终一致性方案。
2. **Caffeine + Redis 多级缓存**：L1 微秒级本地缓存 + L2 Redis 分布式缓存，写时双删保证一致性。
3. **Sentinel 限流降级**：QPS 限流 + 热点参数限流 + 异常熔断，防止系统被流量打垮。
4. **可靠消息传递**：Publisher Confirm + Outbox + 重试队列 + 死信队列，多层保障消息不丢失。
5. **容器化部署**：Docker Compose 一键启动全部中间件 + 应用，开箱即用。

**难点：**
1. **多级缓存一致性**：L1(Caffeine) 与 L2(Redis) 的一致性如何保证？→ 写时双删 + 短 TTL 兜底。
2. **Sentinel 规则调优**：限流阈值如何确定？→ 压测确定系统水位线，逐步调优。热点参数限流防止单券打垮全局。
3. **消息可靠性**：MQ 发送失败如何保证不丢消息？→ Outbox 模式 + 定时补偿。
4. **消费失败处理**：消费异常如何恢复？→ 重试队列延时重试 + 死信队列回滚 Redis 数据。
5. **幂等与一人一单**：集群模式下 `synchronized` 失效 → Redisson 分布式锁 + 数据库去重双重保障。
